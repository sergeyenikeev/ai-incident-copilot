# Архитектурные решения

## Зачем нужен этот документ

Этот файл фиксирует ключевые архитектурные решения проекта в человекочитаемой
форме. Его задача — объяснить не только «что сделано», но и «почему выбрали
именно этот путь».

Если начинать с самого важного, то проект строится вокруг пяти опорных решений:

1. быстрый API и асинхронный анализ
2. event journal в БД
3. отдельный workflow-tracing слой
4. строгая наблюдаемость
5. инфраструктура, готовая к постепенному усложнению

## Решение 1. Анализ вынесен из API в worker

### Контекст

Анализ инцидента может быть относительно дорогим:

- несколько шагов обработки
- обращение к БД
- возможная будущая LLM-логика

Если делать это прямо в HTTP-запросе, API станет медленным и хрупким.

### Решение

API:

- принимает запрос
- сохраняет инцидент
- публикует событие

Worker:

- читает событие
- запускает workflow
- сохраняет результат

### Почему это хорошо

- HTTP-ответ остаётся быстрым
- worker можно масштабировать отдельно
- анализ не блокирует клиентский трафик

### Trade-off

- система становится более сложной
- появляется необходимость в event journal и идемпотентности

## Решение 2. Использовать `incident_events` как event journal и outbox-подобный слой

### Контекст

Если просто писать инцидент в БД и сразу отправлять Kafka-сообщение, то при
ошибке между этими двумя шагами легко получить потерю события.

### Решение

Перед публикацией событие сначала сохраняется в `incident_events`.

Только после этого выполняется отправка в Kafka.

### Почему это хорошо

- событие не теряется при временной недоступности Kafka
- можно хранить `retry_count`, `last_error`, `status`
- worker и API имеют единое место правды о судьбе события

### Trade-off

- нужно отдельно синхронизировать статусы `pending/published/failed/consumed`
- это не полноценный distributed transaction

## Решение 3. Разделить `workflow_runs` и `workflow_steps`

### Контекст

Если хранить только общий факт анализа, то невозможно качественно понять,
где именно сломался workflow.

### Решение

Сделаны две сущности:

- `workflow_runs` — запуск целиком
- `workflow_steps` — отдельные node execution

### Почему это хорошо

- легко видеть общий статус анализа
- легко диагностировать конкретный шаг
- удобно строить audit и operational-отчёты

### Trade-off

- больше таблиц и больше записей
- немного больше сложности в persistence-слое

## Решение 4. Сохранить analyzer отдельно от orchestration

### Контекст

Логика анализа может измениться:

- сегодня rule-based
- завтра LLM
- позже гибридная модель

Если смешать алгоритм анализа с persistence и LangGraph-обвязкой, систему
будет трудно эволюционировать.

### Решение

Разделить:

- `rule_based.py` — алгоритм
- `service.py` — orchestration и persistence
- `state.py` — контракт состояния workflow

### Почему это хорошо

- analyzer можно заменить без переписывания storage-слоя
- тесты становятся проще
- orchestration остаётся стабильным

### Trade-off

- чуть больше файлов и абстракций

## Решение 5. Использовать `NoOpEventPublisher`

### Контекст

Тесты и часть локальных сценариев не должны зависеть от реального Kafka broker.

### Решение

Ввести единый контракт `EventPublisher` и две реализации:

- `KafkaEventPublisher`
- `NoOpEventPublisher`

### Почему это хорошо

- прикладной код не обрастает `if kafka_enabled`
- тесты легче и быстрее
- локальная разработка проще

### Trade-off

- часть сценариев тестируется без живого брокера

## Решение 6. Структурированное JSON-логирование с contextvars

### Контекст

В асинхронной системе с API, worker, БД и Kafka обычные строковые логи быстро
становятся почти бесполезными.

### Решение

Использовать:

- `structlog`
- JSON-формат
- `request_id`
- `incident_id`
- `workflow_run_id`
- `event_id`

### Почему это хорошо

- можно коррелировать события между слоями
- проще строить observability и поиск по логам
- логи становятся пригодными для production

### Trade-off

- логирование чуть сложнее в bootstrap

## Решение 7. Health endpoint возвращает `degraded`, а не только `ok/fail`

### Контекст

Сервис может быть жив как процесс, но иметь частичную деградацию зависимостей.

### Решение

`/health` агрегирует:

- API
- БД
- Kafka publisher

И в случае частичной проблемы отдаёт `degraded`.

### Почему это хорошо

- оператор видит более точную картину
- проще отличать полное падение от частичной деградации

### Trade-off

- нужно договориться, как именно оркестратор и мониторинг трактуют degraded

## Решение 8. Kubernetes-манифесты используют отдельный migration Job

### Контекст

Запуск `alembic upgrade head` внутри каждого API-pod приводит к неустойчивому
rollout и смешивает две разные ответственности:

- применение схемы БД
- запуск приложения

### Решение

Сделать отдельный migration Job:

- plain manifests: `k8s/migration-job.yaml`
- Helm: hook Job `pre-install,pre-upgrade`

### Почему это хорошо

- миграции выполняются отдельно
- API-поды стартуют чище
- rollout становится предсказуемее

### Trade-off

- в deployment-процессе появляется дополнительный объект

## Решение 9. Worker readiness через файл, а не HTTP endpoint

### Контекст

Worker не обслуживает HTTP-трафик, но Kubernetes всё равно нужен сигнал
готовности процесса.

### Решение

`worker_main.py` создаёт readiness-файл после старта зависимостей, а probes
проверяют его наличие.

### Почему это хорошо

- не нужно искусственно придумывать HTTP-сервер для worker
- сигнал готовности связан с фактическим bootstrap-процессом

### Trade-off

- схема менее привычна, чем HTTP health endpoint

## Решение 10. Тесты используют SQLite, но интеграционные сценарии всё равно гоняют миграции

### Контекст

Нужен быстрый тестовый контур, но без отрыва от реальной схемы БД.

### Решение

Использовать SQLite:

- для скорости
- для изоляции
- для удобства CI

Но при этом в интеграционных тестах применять реальные Alembic-миграции.

### Почему это хорошо

- тесты быстрые
- схема проверяется ближе к реальности
- код не теряет production-форму

### Trade-off

- SQLite и PostgreSQL всё же не идентичны

## Решение 11. Ответы API заворачиваются в envelope

### Контекст

Нужно единообразие контрактов и возможность эволюции ответов.

### Решение

Использовать:

- `ResponseEnvelope`
- `PaginatedResponse`
- `ErrorEnvelope`

### Почему это хорошо

- у клиентов предсказуемая структура
- метаданные можно добавлять без ломки payload
- ошибки единообразны

### Trade-off

- ответ чуть более вложенный, чем «плоский JSON»

## Что это всё даёт вместе

В сумме архитектура проекта решает три базовые production-задачи:

1. безопасно принимать данные
2. асинхронно и наблюдаемо их обрабатывать
3. оставлять полный след в БД и логах

Именно поэтому проект выглядит более многослойным, чем «обычный CRUD-сервис».

## Когда эти решения стоит пересматривать

Пересмотр может понадобиться, если:

- резко вырастет нагрузка на Kafka и worker
- появится real-time SLA на анализ
- потребуется полноценный outbox relay
- понадобится строгий multi-tenant режим
- возникнет требование на DLQ и replay pipeline

Но на текущем этапе эти решения хорошо соответствуют задачам проекта и его
production-style целям.
