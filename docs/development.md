# Описание разработки

## Зачем нужен этот документ

Этот файл объясняет не только то, что находится в репозитории, но и почему
проект был построен именно так.

Если начинать с самого важного, то архитектура опирается на три идеи:

1. HTTP-запрос должен быть коротким и предсказуемым.
2. Длительная обработка должна жить в отдельном worker-процессе.
3. Каждое важное действие должно оставлять след в БД и логах.

Из этих трёх принципов выросла вся остальная структура проекта.

## Главная цель проекта

`ai-incident-copilot` нужен для автоматизированной обработки инцидентов:

- принять сигнал об инциденте
- сохранить его в БД
- поставить в event-driven pipeline
- проанализировать через LangGraph
- выдать классификацию, severity и рекомендацию
- сохранить историю выполнения

Проект не сводится только к "FastAPI + моделька". Он специально сделан как
production-style backend, где важны наблюдаемость, повторяемость, идемпотентность
и готовность к эксплуатации.

## Почему архитектура разделена на слои

### Почему есть слой `api`

API должно решать только transport-задачи:

- принять запрос
- провалидировать его
- вернуть корректный HTTP-ответ

Если бы бизнес-логика жила прямо в роутерах, код быстро стал бы хрупким:
труднее тестировать, сложнее переиспользовать и почти невозможно масштабировать.

### Почему есть слой `application`

Слой `application` содержит прикладные сценарии:

- создание инцидента
- постановка в анализ
- выполнение workflow

Он знает правила системы, но не зависит от HTTP-формата ответа.

### Почему есть слой `db/repositories`

Репозитории нужны, чтобы SQLAlchemy-запросы не были размазаны по сервисам.

Это даёт:

- более чистые сервисы
- проще тестировать
- проще менять стратегию запросов и индексы

### Почему есть отдельный слой `events`

Kafka-интеграция отделена от сервиса инцидентов, потому что публикация и
чтение событий — это инфраструктурная задача, а не предметная логика.

### Почему worker вынесен в отдельный процесс

Анализ инцидента может быть тяжёлым и потенциально медленным, особенно если
потом вместо rule-based логики появится LLM.

Если выполнять анализ прямо в HTTP-запросе:

- возрастёт latency
- появится риск таймаутов
- API станет хуже масштабироваться

Поэтому анализ вынесен в отдельный worker, который читает Kafka-события.

## Как проект собирался по этапам

### 1. Базовый каркас репозитория

Сначала был собран фундамент:

- `pyproject.toml`
- зависимости через `uv`
- структура `src/`
- структура `tests/`
- Alembic

Задача этого шага — задать единый, воспроизводимый скелет проекта.

### 2. Конфигурация и логирование

Дальше были добавлены:

- `core/config.py`
- `core/logging.py`

Почему это делалось рано:

- почти все остальные модули опираются на настройки
- structured logging нужен с самого начала, иначе потом тяжело унифицировать логи

### 3. Модель данных и миграции

Следующий шаг — проектирование таблиц:

- `incidents`
- `workflow_runs`
- `workflow_steps`
- `incident_events`
- `audit_logs`

Логика проектирования была такой:

- `incidents` хранит бизнес-сущность
- `workflow_runs` хранит факт запуска анализа
- `workflow_steps` хранит поузловую историю LangGraph
- `incident_events` хранит event-journal и статус публикации/обработки
- `audit_logs` хранит внутренние действия системы

Это позволило сразу заложить наблюдаемость и retry-safe модель.

### 4. REST API

После слоя хранения был добавлен HTTP-вход через FastAPI.

Почему после БД:

- API без модели хранения быстро приводит к временным костылям
- правильнее сначала понять, какие сущности и статусы живут в системе

Что было добавлено:

- versioned routes `/api/v1`
- Pydantic v2 схемы
- единые envelopes ответов
- валидация
- structured error handling

### 5. Прикладной сервис инцидентов

`IncidentService` появился как граница между API и остальными слоями.

Почему он нужен:

- роутер не должен знать о транзакциях
- роутер не должен знать о Kafka и outbox-модели
- прикладной сценарий должен быть переиспользуемым и тестируемым отдельно

Именно здесь была реализована идемпотентность создания инцидента.

### 6. Event-driven связка через Kafka

После появления API и прикладного слоя был добавлен Kafka publisher.

Ключевая идея:

- сначала сохранить событие в БД
- потом отправить его в Kafka

Это не полноценный distributed transaction, но практичная outbox-подобная
модель, которая защищает от потери бизнес-события.

### 7. LangGraph workflow

Когда канал событий был готов, был добавлен workflow-слой.

Почему выбран LangGraph:

- он естественно подходит для stateful pipeline
- позволяет явно описать граф обработки
- облегчает будущую замену rule-based узлов на LLM

Почему workflow разделён на `state`, `service` и `rule_based`:

- `state.py` фиксирует контракт состояния
- `service.py` отвечает за orchestration и persistence
- `rule_based.py` отвечает только за алгоритм

Такой разрез сильно уменьшает связанность кода.

### 8. Worker

После workflow появился отдельный worker.

Что он делает:

- читает `incident.analysis.requested`
- не допускает повторной обработки уже завершённого события
- делает локальные retry
- запускает workflow
- публикует `incident.analysis.completed`

Почему offset'ы коммитятся вручную:

- нужен контроль над тем, в какой момент сообщение считается обработанным
- нужен предсказуемый retry-flow

### 9. Тесты

Тестирование добавлялось как отдельный этап, но фактически влияло на дизайн
всего проекта.

Что покрыто:

- unit-тесты rule-based логики
- unit-тесты сервисов
- тесты Kafka-обвязки
- интеграционные тесты API
- интеграционный сценарий worker-flow

Почему это важно:

- проект проверяет не только happy-path
- видно, что idempotency и workflow действительно работают
- entrypoint'ы и degraded-сценарии тоже верифицированы

Отдельно важно, что тестовый контур использует SQLite и фикстуры, но при этом
runtime-код остаётся ориентирован на PostgreSQL. Это даёт быстрые тесты без
потери архитектурной формы production-кода.

### 10. Docker и Docker Compose

Контейнеризация была добавлена после того, как локальный runtime стал
достаточно стабильным.

Что это дало:

- воспроизводимый локальный запуск
- единый runtime-образ для API и worker
- удобный dev/demo контур

### 11. CI/CD

GitHub Actions нужны для автоматического quality gate:

- Ruff
- Mypy
- Pytest
- coverage
- сборка Docker-образов

Это защищает проект от деградации качества при дальнейшем развитии.

Почему pipeline разделён на `quality` и `docker`:

- сначала проект должен доказать корректность кода
- только потом имеет смысл тратить время на сборку и публикацию образов

### 12. Kubernetes и Helm

Когда контейнерный runtime был готов, были добавлены Kubernetes manifests и
Helm chart.

Отдельно важное решение:

- миграции БД вынесены в отдельный Job

Почему это важно:

- API-pod не должен гонять `alembic upgrade head` при каждом рестарте
- миграции становятся более предсказуемыми при rollout

Дополнительно были добавлены readiness/liveness probes и отдельный readiness-файл
для worker-процесса, потому что у worker нет HTTP endpoint'а, но Kubernetes всё
равно должен понимать, что процесс реально готов к работе.

### 13. Документация и комментарии

Финальный этап — сделать проект удобным не только для запуска, но и для
сопровождения:

- README
- архитектурные документы
- описание API
- описание workflow
- комментарии в ключевых модулях
- модульный гид

Позже этот слой был расширен ещё сильнее:

- подробные комментарии в support-модулях
- документ по модулям проекта
- документ по ходу разработки и архитектурным решениям
- пояснения по инфраструктурным артефактам

## Самые важные архитектурные решения

### Идемпотентность создания инцидента

Решение:

- использовать `Idempotency-Key`
- хранить его в `incidents.idempotency_key`

Зачем:

- HTTP-клиент может повторить запрос
- система не должна создавать дубликаты при retry

### Журнал событий в `incident_events`

Решение:

- писать доменное событие в БД до публикации в Kafka

Зачем:

- не терять событие при временной недоступности Kafka
- иметь единое место для статуса `pending/published/failed/consumed`

### Разделение `workflow_runs` и `workflow_steps`

Решение:

- хранить запуск workflow отдельно от шагов

Зачем:

- видеть и общий статус анализа, и детальную историю узлов
- удобно расследовать, где именно произошёл сбой

### Отдельный `AuditLog`

Решение:

- не смешивать аудит действий приложения с Kafka-событиями

Зачем:

- это разные сущности с разными сценариями чтения

### `NoOpEventPublisher`

Решение:

- дать тестам и локальному окружению реализацию publisher без реального Kafka

Зачем:

- минимизировать количество условной логики в прикладном коде
- упростить тестирование

Подробный список решений и trade-off'ов: `docs/decisions.md`

## Что можно развивать дальше

### Замена rule-based анализа на LLM

Для этого уже есть хороший задел:

- workflow-state типизирован
- orchestration вынесен отдельно
- analyzer изолирован отдельным модулем

Фактически нужно заменить только сам алгоритм анализа, сохранив контракт.

### Расширение workflow

Можно добавлять:

- новые узлы графа
- новые ветки
- обогащение через внешние источники
- шаги подтверждения человеком

### Усиление event-пайплайна

Возможные улучшения:

- отдельный outbox relay
- DLQ
- более сложная стратегия backoff
- повторная отправка failed-событий по cron/worker-job

## Как объяснять проект новому разработчику

Самая полезная краткая формулировка такая:

`ai-incident-copilot` — это backend, в котором API быстро принимает инцидент,
фиксирует его в БД и ставит в event-driven pipeline, а отдельный worker через
LangGraph анализирует инцидент, сохраняет все шаги и возвращает результат
обратно в доменную модель и Kafka.

Если человек понял эту фразу, дальше уже можно разбирать детали модулей.
