# Runbook и сопровождение

## Зачем нужен этот документ

Этот файл нужен для дежурного инженера или разработчика, который должен быстро
понять, что делать при типовых operational-сценариях.

Он не заменяет мониторинг и алерты, но даёт краткий человеческий план действий.

## Самое важное

Если сервис ведёт себя странно, порядок проверки такой:

1. посмотреть `/health`
2. посмотреть логи API или worker
3. проверить PostgreSQL
4. проверить Kafka
5. проверить таблицы `incident_events`, `workflow_runs`, `workflow_steps`

В большинстве случаев проблема локализуется уже на этих пяти шагах.

## Сценарий 1: API отвечает ошибками

### Что проверить

- доступен ли `/health`
- есть ли `request_id` в логах
- не деградировал ли PostgreSQL
- не появились ли массовые validation errors или internal errors

### Где смотреть

- логи API
- `audit_logs`
- `incident_events`

### Что это обычно означает

- ошибка входных данных клиента
- проблема БД
- деградация Kafka publisher

## Сценарий 2: инциденты создаются, но анализ не выполняется

### Что проверить

- есть ли событие `incident.analysis.requested` в `incident_events`
- в каком статусе это событие: `pending`, `failed`, `consumed`
- жив ли worker
- существует ли readiness-файл worker
- есть ли записи в `workflow_runs`

### Интерпретация

Если `incident.analysis.requested` вообще не создаётся:

- проблема в API или `IncidentService`

Если событие есть, но worker его не потребляет:

- проблема в worker или Kafka consumer

Если `workflow_runs` создаются, но не завершаются:

- проблема внутри workflow

## Сценарий 3: Kafka недоступна

### Как это проявляется

- `/health` возвращает `degraded`
- в логах появляются ошибки publisher/consumer
- события в `incident_events` получают статус `failed`

### Что делать

1. проверить доступность broker и bootstrap servers
2. проверить сетевую связность между pod/container и Kafka
3. проверить аутентификацию, если она появится позже
4. после восстановления оценить, нужна ли повторная отправка failed-событий

### Важно

Сам инцидент при этом не теряется, потому что проект сначала сохраняет его в БД.

## Сценарий 4: worker жив, но часть сообщений падает

### Что проверить

- `incident_events.last_error`
- `incident_events.retry_count`
- `workflow_runs.error_message`
- `workflow_steps.error_message`

### Что это даёт

Можно понять:

- на каком именно событии всё ломается
- на каком шаге workflow произошёл сбой
- была ли проблема разовой или повторялась несколько раз

## Сценарий 5: миграции не применяются

### Для Docker Compose

Проверить:

- стартовал ли контейнер API
- нет ли ошибок Alembic в логе контейнера
- доступен ли PostgreSQL

### Для Kubernetes

Проверить:

- статус `migration-job`
- логи job
- корректность `Secret` и `ConfigMap`
- доступность БД из namespace

## Полезные таблицы для диагностики

### `incidents`

Показывает текущее бизнес-состояние инцидента:

- создан
- поставлен в анализ
- анализируется
- проанализирован
- упал

### `incident_events`

Показывает техническую судьбу событий:

- созданы ли они
- опубликованы ли
- обработаны ли
- сколько было retry

### `workflow_runs`

Показывает факт запуска анализа и общий итог.

### `workflow_steps`

Показывает конкретный шаг графа, на котором всё пошло не так.

### `audit_logs`

Полезна для бизнес-аудита и расследования действий API.

## Какие логи особенно важны

### Для API

Ищите поля:

- `request_id`
- `incident_id`
- `error_code`
- `status_code`

### Для worker

Ищите поля:

- `incident_id`
- `event_id`
- `workflow_run_id`
- `attempt`

## Что делать после восстановления

После устранения проблемы полезно:

1. проверить `/health`
2. убедиться, что новые инциденты проходят полный цикл
3. посмотреть, остались ли `failed` события
4. решить, нужна ли ручная re-drive стратегия для этих событий

## Что можно улучшить дальше

Если проект будет развиваться в production глубже, следующим логичным шагом будет:

- автоматический re-drive failed-событий
- DLQ
- отдельные operational dashboards
- алерты по stuck workflow и failed event ratio

Но даже в текущем виде runbook уже даёт хороший базовый сценарий сопровождения.
