# Модель данных

## Зачем нужен этот документ

Этот файл подробно объясняет, как устроено хранение данных в проекте:

- какие таблицы существуют
- зачем нужна каждая таблица
- как таблицы связаны друг с другом
- какие поля являются критичными для runtime

Если читать проект с точки зрения постоянного состояния системы, это один из
самых важных документов.

## Самая важная идея

В проекте есть не одна «главная таблица», а сразу несколько слоёв хранения:

1. `incidents` — бизнес-сущность
2. `incident_events` — технический журнал событий
3. `workflow_runs` — факт запуска анализа
4. `workflow_steps` — детальная история шагов анализа
5. `audit_logs` — аудит действий системы

Это разделение сделано осознанно. Одна таблица не может одинаково хорошо
решать задачи бизнес-состояния, event journal, workflow tracing и аудита.

## Таблица `incidents`

### Для чего нужна

Это главная бизнес-таблица проекта. Она отвечает на вопрос:

«Какой инцидент сейчас существует в системе и каков его текущий итоговый статус?»

### Что в ней хранится

- `id` — идентификатор инцидента
- `title` — краткий заголовок
- `description` — подробное описание
- `source` — источник сигнала
- `status` — состояние инцидента
- `classification` — итоговая категория анализа
- `severity` — уровень критичности
- `priority_score` — численная оценка приоритета
- `recommendation` — финальная рекомендация
- `last_analysis_at` — время последнего анализа
- `idempotency_key` — защита от повторного создания
- `metadata` — дополнительные данные от источника
- `created_at`, `updated_at`

### Почему `idempotency_key` хранится здесь

Потому что именно на уровне бизнес-сущности нужно решить проблему повторного
`POST /api/v1/incidents`.

Если один и тот же клиентский запрос придёт повторно:

- инцидент не должен создаться заново
- система должна вернуть уже существующий объект

## Таблица `incident_events`

### Для чего нужна

Это event journal и outbox-подобный слой проекта.

Она отвечает на вопрос:

«Какие доменные события были созданы, опубликованы, потреблены или завершились ошибкой?»

### Что в ней хранится

- `id` — идентификатор события
- `incident_id` — связь с инцидентом
- `event_type` — тип события
- `kafka_topic` — topic публикации
- `event_key` — Kafka key
- `status` — технический статус события
- `idempotency_key` — ключ повторной обработки
- `payload` — JSON payload события
- `retry_count` — число повторов
- `processed_at` — время успешной публикации или потребления
- `last_error` — последняя ошибка
- `created_at`, `updated_at`

### Почему это отдельная таблица

Потому что событие — это не то же самое, что инцидент.

Инцидент живёт как бизнес-сущность, а событие живёт как факт интеграции между
подсистемами. У события есть собственный жизненный цикл:

- `pending`
- `published`
- `consumed`
- `failed`

Этот жизненный цикл нельзя корректно хранить только в `incidents`.

### Почему `idempotency_key` совпадает с `event_id`

Это делает worker-логику проще и надёжнее:

- событие можно искать по стабильному ключу
- повторная обработка определяется однозначно
- источник идемпотентности совпадает с метаданными самого event

## Таблица `workflow_runs`

### Для чего нужна

Эта таблица хранит один запуск анализа целиком.

Она отвечает на вопрос:

«Был ли запущен workflow по конкретному инциденту, чем он закончился и с каким входом?»

### Что в ней хранится

- `id`
- `incident_id`
- `trigger_event_id`
- `workflow_name`
- `status`
- `input_payload`
- `output_payload`
- `error_message`
- `started_at`
- `finished_at`
- `created_at`, `updated_at`

### Почему запуск анализа вынесен отдельно от `incidents`

Потому что один инцидент потенциально может анализироваться больше одного раза:

- повторный анализ
- повторная обработка после сбоя
- будущие расширения workflow

Если бы всё это хранилось прямо в `incidents`, история запусков терялась бы.

## Таблица `workflow_steps`

### Для чего нужна

Хранит детальную историю выполнения LangGraph по шагам.

Она отвечает на вопрос:

«Какой именно node выполнился, с какими входными данными и на каком шаге возникла ошибка?»

### Что в ней хранится

- `id`
- `workflow_run_id`
- `node_name`
- `status`
- `input_payload`
- `output_payload`
- `error_message`
- `started_at`
- `finished_at`
- `created_at`

### Почему нельзя ограничиться только `workflow_runs`

Потому что общий статус запуска недостаточен для диагностики.

Если хранить только run:

- непонятно, какой именно node упал
- непонятно, на каком шаге произошло ветвление
- невозможно качественно расследовать поведение workflow

## Таблица `audit_logs`

### Для чего нужна

Хранит внутренний аудит действий приложения.

Она отвечает на вопрос:

«Какие важные действия были выполнены системой и в рамках какого запроса?»

### Что в ней хранится

- `id`
- `incident_id`
- `entity_type`
- `entity_id`
- `action`
- `actor`
- `request_id`
- `payload`
- `created_at`

### Почему аудит не хранится в `incident_events`

Потому что это разные сущности:

- `incident_events` нужны для интеграции и event-driven обработки
- `audit_logs` нужны для внутренней трассировки действий приложения

Они читаются по-разному и отвечают на разные operational-вопросы.

## Связи между таблицами

### `incidents -> incident_events`

Один инцидент может породить много событий.

Примеры:

- `incident.created`
- `incident.analysis.requested`
- `incident.analysis.completed`

### `incidents -> workflow_runs`

Один инцидент может иметь несколько запусков workflow.

### `workflow_runs -> workflow_steps`

Один запуск workflow содержит много шагов.

### `incident_events -> workflow_runs`

Поле `trigger_event_id` связывает запуск workflow с событием, которое его
инициировало.

### `incidents -> audit_logs`

Один инцидент может иметь много записей аудита.

## Какие поля особенно критичны

### В `incidents`

- `status`
- `classification`
- `severity`
- `idempotency_key`

### В `incident_events`

- `event_type`
- `status`
- `idempotency_key`
- `retry_count`
- `last_error`

### В `workflow_runs`

- `status`
- `trigger_event_id`
- `error_message`

### В `workflow_steps`

- `node_name`
- `status`
- `error_message`

### В `audit_logs`

- `action`
- `request_id`
- `payload`

## Как меняются данные по жизненному циклу

### Сценарий создания инцидента

1. в `incidents` появляется новая запись
2. в `audit_logs` появляется `incident.created`
3. в `incident_events` появляется `incident.created`
4. статус события становится `published` или `failed`

### Сценарий запроса анализа

1. `incidents.status` меняется на `analysis_requested`
2. в `audit_logs` появляется `incident.analysis_requested`
3. в `incident_events` появляется `incident.analysis.requested`

### Сценарий успешного анализа

1. worker читает `incident.analysis.requested`
2. `workflow_runs` получает новую запись
3. `workflow_steps` получает записи по шагам
4. `incidents` обновляется итогом анализа
5. исходное событие получает `consumed`
6. создаётся `incident.analysis.completed`

### Сценарий ошибки анализа

1. `workflow_run` получает `failed`
2. `workflow_steps` фиксируют конкретный сбойный шаг
3. `incidents.status` становится `failed`
4. `incident_events` обновляет `retry_count` и `last_error`

## Почему везде есть timestamps

Временные метки нужны для:

- аудита
- построения отчётов
- расследования инцидентов
- отладки retry-сценариев
- operational-мониторинга

Без них почти невозможно понять последовательность событий в асинхронной системе.

## Почему JSON-поля допустимы в этой модели

Поля `metadata`, `payload`, `input_payload`, `output_payload` сделаны JSON,
потому что они хранят:

- вариативные контексты инцидента
- Kafka payload
- workflow state

Строго нормализовывать все эти данные в отдельные таблицы на старте проекта
было бы слишком дорого и сильно усложнило бы развитие.

При этом важные queryable-поля всё равно вынесены в отдельные колонки:

- `status`
- `classification`
- `severity`
- `event_type`
- `node_name`

## Что можно развивать дальше

Если система станет заметно крупнее, возможны следующие эволюции:

- отдельная таблица для replay failed events
- таблица для operator actions/manual override
- партиционирование `audit_logs` и `incident_events`
- secondary indexes под аналитические отчёты
- soft-delete или архивирование старых workflow-артефактов

Но в текущем проекте модель уже хорошо сбалансирована между простотой и
production-полезностью.
