# Тестирование проекта

## Зачем нужен этот документ

Этот файл объясняет, как устроен тестовый контур проекта, какие слои он
покрывает и почему тесты написаны именно так.

Если начинать с самого важного, тестовая стратегия у проекта такая:

1. быстро проверять бизнес-логику в unit-тестах
2. проверять сквозные сценарии через интеграционные тесты
3. держать контур изолированным и воспроизводимым

## Главная идея тестового контура

Тесты должны подтверждать не только «код запускается», но и то, что архитектура
реально работает:

- API валидирует запросы
- БД хранит сущности и статусы
- Kafka-обвязка не ломает прикладной слой
- worker корректно обрабатывает события
- workflow сохраняет run и steps

Поэтому тесты покрывают не только чистые функции, но и стыки между слоями.

## Почему используется SQLite

В тестах проект использует временную SQLite-базу.

Это решение принято для скорости и изоляции:

- тесты запускаются быстро
- не нужен внешний PostgreSQL
- каждый тестовый запуск получает чистую БД

При этом важно, что:

- runtime-код по-прежнему ориентирован на PostgreSQL
- Alembic-миграции всё равно прогоняются в интеграционных сценариях

Такой баланс даёт быстрый feedback loop и сохраняет форму production-кода.

## Как устроены фикстуры

### `tests/conftest.py`

Это главный bootstrap тестового окружения.

Что делает:

- создаёт временный файл SQLite
- строит sync и async DSN
- применяет Alembic-миграции
- создаёт `Settings` для тестов
- поднимает `FastAPI` через `create_app`
- создаёт `TestClient`
- даёт лёгкий `DatabaseManager` для unit-тестов

Почему это важно:

- тесты остаются короткими
- setup не дублируется по файлам
- сценарии ближе к реальному runtime

## Структура тестов

### `tests/unit/`

Здесь лежат быстрые и изолированные тесты.

#### `test_rule_based_workflow.py`

Проверяет rule-based анализатор:

- классификацию
- severity
- маршрутизацию
- генерацию рекомендаций

#### `test_incident_service.py`

Проверяет прикладной сервис:

- создание инцидента
- идемпотентность
- постановку в анализ

#### `test_kafka_components.py`

Проверяет Kafka-обвязку без реального Kafka broker.

#### `test_workflow_service.py`

Проверяет, что workflow:

- создаёт `workflow_runs`
- создаёт `workflow_steps`
- обновляет инцидент

#### `test_entrypoints.py`

Проверяет bootstrap-логику:

- запуск Uvicorn для API
- lifecycle worker entrypoint
- readiness-файл

### `tests/integration/`

Здесь лежат более широкие сценарии.

#### `test_api.py`

Проверяет:

- `/health`
- создание инцидента
- получение по id
- список с фильтрацией
- структурированные ошибки

#### `test_worker_flow.py`

Проверяет сквозной worker-сценарий:

- API создаёт событие запроса анализа
- worker обрабатывает payload
- инцидент получает итог анализа
- появляются `workflow_runs`, `workflow_steps`
- исходное событие помечается как `consumed`
- создаётся `incident.analysis.completed`

## Почему есть и unit-, и integration-тесты

### Unit-тесты нужны для:

- быстрого feedback loop
- проверки чистой логики
- локализации ошибок

### Интеграционные тесты нужны для:

- проверки совместной работы слоёв
- проверки схемы БД и миграций
- проверки того, что реальные контракты между модулями совпадают

Без unit-тестов сложно быстро развивать код.  
Без integration-тестов легко получить «по отдельности всё работает», но система
ломается на стыках.

## Что именно считается важным покрытием

Самые критичные зоны покрытия:

- `incident_service`
- `workflow_service`
- `worker_main`
- `health`
- Kafka publisher/consumer abstractions

Это те места, где сходятся несколько подсистем и где ошибка обычно стоит дороже.

## Почему часть сценариев тестируется без реального Kafka

Полноценный broker сильно усложнил бы локальный и CI-контур.

Вместо этого проект использует:

- `NoOpEventPublisher`
- mock payload
- прямой вызов worker-логики

Это даёт:

- быстрые тесты
- предсказуемый результат
- достаточное покрытие orchestration-слоя

При необходимости позднее можно добавить отдельный docker-based e2e слой.

## Как запускать тесты

Базовый запуск:

```bash
uv run pytest
```

Линтер и типы перед тестами:

```bash
uv run ruff check src alembic tests
uv run mypy src
```

## Как расширять тестовый контур

Если добавляется новый прикладной сценарий, лучше идти так:

1. unit-тест на чистую бизнес-логику
2. integration-тест на стык слоёв
3. при необходимости entrypoint/infra-тест на bootstrap-поведение

Если добавляется новый node в workflow:

1. покрыть analyzer-логику unit-тестом
2. проверить persistence шага через workflow service
3. при необходимости расширить worker-flow сценарий

## Что можно улучшить дальше

Потенциальные направления:

- отдельные e2e-тесты с реальным Kafka и PostgreSQL через Docker
- smoke-тесты для Helm/Kubernetes manifests
- contract-тесты на формат Kafka-событий
- отдельные regression-тесты на retry/backoff сценарии

Но уже сейчас тестовый контур достаточно силён для уверенного развития проекта.
